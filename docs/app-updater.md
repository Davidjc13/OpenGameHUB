# App updater

## Goal

Let installations published on [GitHub Releases](https://github.com/Davidjc13/OpenGameHUB/releases) update without manually downloading from the site.

## Component

File: `Services/AppUpdateService.cs`

Model: `AppReleaseInfo` (tag, HTML URL, download URL, asset name, size).

## Current version

`AppUpdateService.CurrentVersion` reads:

1. `AssemblyInformationalVersionAttribute` (without `+commit` suffix)
2. Fallback: three-component `AssemblyVersion`
3. Development: `dev` by default in `.csproj`

Release builds embed the version via `build-installer.ps1`:

```powershell
-p:InformationalVersion=alpha-0.0.7
-p:Version=0.0.7
```

Only **InformationalVersion** carries the `alpha-` prefix; `AssemblyVersion` uses numeric semver for .NET compatibility.

## Check for updates

### API

```
GET https://api.github.com/repos/Davidjc13/OpenGameHUB/releases?per_page=30
```

**Why not `/releases/latest`:** current releases are `prerelease: true` (`alpha-*`). GitHub's `latest` endpoint **ignores** prereleases and would return 404.

**Why not `per_page=1`:** the API returns releases in **creation order**, not version order. A hotfix tag published after a newer version would be picked incorrectly. The app scans up to 30 releases and picks the **highest version** that has an installer asset.

### Expected asset

Name: `OpenGameHUB-Setup-*.exe` (generated by Inno Setup in CI).

### Version comparison

`ReleaseVersionComparer` in the same file:

- Parses `alpha-`, `beta-`, or stable prefixes
- Compares numeric `System.Version` (e.g. `0.0.10` > `0.0.9`)
- Handles build suffixes after a dash (`alpha-0.0.9-1` → version `0.0.9`, suffix `1`)
- If channels differ, orders alpha < beta < stable

`dev` build: `IsNewer` is always true if a release exists (allows testing in Settings).

## UI flows

### On startup (`MainWindowViewModel`)

```
CheckForAppUpdateAsync (immediate)
StartPeriodicAppUpdateChecks → every 2 hours
  if IsDevBuild → skip
  if newer release → banner + StatusText hint
```

### Background interval

`AppUpdateService.BackgroundCheckInterval` = **2 hours** (`PeriodicTimer`).

When a newer release exists:

- Blue **banner** on the main window with version tag
- **Install update** — downloads and runs the Inno installer (same as Settings)
- **Later** — hides the banner until the next check (or until a different tag appears)

Status bar still shows `AppUpdateAvailableHint` for 12 seconds.

### In Settings (`SettingsViewModel`)

| Action | Behavior |
|--------|----------|
| On open | Automatic `CheckForUpdatesAsync` |
| Check for updates | Same manual check |
| Download and install | `DownloadInstallerAsync` + `LaunchInstallerAndExit` |

Progress bar: indeterminate when querying API; percentage during download.

## Download

- Destination: `%TEMP%\OpenGameHUB\updates\{assetName}`
- If the file exists, tries to delete it; on failure, uses a GUID-suffixed name
- `HttpClient` with User-Agent `OpenGameHUB-Updater/1.0`, 10 min timeout

## Installation

The app exits and a detached `.cmd` helper runs the installer, then relaunches OpenGameHUB:

```batch
installer.exe /SILENT /CLOSEAPPLICATIONS
if exist "OpenGameHUB.exe" start "" /D "{app dir}" "OpenGameHUB.exe"
```

Launched via `cmd /c start /MIN` so the helper survives `Environment.Exit(0)`.

Inno Setup (`installer/OpenGameHUB.iss`):

- Same `AppId` across versions → in-place upgrade
- `PrivilegesRequired=lowest` — no admin
- Default directory: `%LocalAppData%\Programs\OpenGameHUB`
- `[Run]` postinstall launches `{app}\OpenGameHUB.exe` with `skipifsilent` (interactive installs only)
- Silent in-app updates rely on the helper script above instead of `[Run]`
- `CloseApplications=force` — unlock files if a stray process remains

**Why a helper script:** the app calls `Environment.Exit` after starting the installer, so Inno's `RestartApplications` has nothing to restart. The `[Run]` postinstall step is also unreliable in `/SILENT` mode, so the helper waits for setup and relaunches explicitly.

**Why exit the app:** the installer must replace the running `OpenGameHUB.exe`; `CLOSEAPPLICATIONS` handles any leftover lock.

## CI

`.github/workflows/build-installer.yml` — two jobs:

- **Build** — every push, pull request, and tag (`dotnet build -c Release`)
- **Publish** — tags only (`alpha-*`, `beta-*`, `x.y.z`): builds the Inno installer and attaches it to the GitHub Release

Tag push → `build-installer.ps1 -AppVersion $tag` → attaches exe to the Release.

The updater and CI assume the **same asset name** and repository `Davidjc13/OpenGameHUB`.

## Known limitations

- Windows only (`.exe` installer)
- Requires internet access to `api.github.com` + `github.com`
- Does not verify digital signature of the installer (trusts HTTPS from official GitHub)
- Does not update legendary or user data — only the OpenGameHUB binary
